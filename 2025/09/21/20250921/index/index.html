性能优化，MyBatis + 多线程 + 流式IO 实战案例






📌 一、问题背景
    

  • 数据量：250万行数据库记录
    • 

  • 需求：每1万条生成1个Excel文件 → 共250个文件
    • 

  • 原始耗时：2小时（120分钟）
    • 

  • 优化目标：4~5分钟内完成
    • 

  • 限制条件：
      

    • 不动任何JVM启动参数（-Xmx等）
      • 

    • 不动数据库配置
      • 

    • 使用 MyBatis
      • 

    • 保证数据完整、线程安全、无内存泄漏
      • 

    

    •

⚙️ 三、完整可运行代码示例





🔍 四、关键优化点详解
1. 真·流式查询（防OOM核心）





❗ 错误认知：很多人以为 fetchSize=10000 是流式 → 实际是预加载！
✅ 正解：MySQL下必须 Integer.MIN_VALUE 才是真流式。






2. SXSSFWorkbook 窗口大小优化





更小的窗口 = 更频繁刷盘 = 更低内存 = 更稳定（尤其服务器环境）






3. 线程池配置（自动限速防崩）





避免任务堆积导致内存爆炸。






4. CountDownLatch 精确控制（确保完成）





比 executor.awaitTermination() 更精确 —— 等的是“任务完成”，不是“线程空闲”。

🧪 六、压测建议（上线前必做）
    

  1. 监控内存：jconsole 或 VisualVM 观察堆内存是否平稳
    2. 

  2. 监控磁盘IO：确保不是磁盘成为瓶颈（考虑用SSD或内存盘/tmp）
    3. 

  3. 异常测试：故意制造一个写入失败，看是否影响其他任务
    4. 

  4. 断点测试：中途kill -STOP再kill -CONT，看是否能恢复
    5.

✅ 八、总结 Checklist
部署前请确认：


    

  •  fetchSize = Integer.MIN_VALUE
    • 

  •  SXSSFWorkbook(100)（窗口100）
    • 

  •  线程池 CallerRunsPolicy
    • 

  •  latch.countDown() 在 finally 块中
    • 

  •  workbook.dispose() 调用
    • 

  •  超时保护 latch.await(timeout)
    • 

  •  异常不中断主线程
    •
